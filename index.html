    Introduction

    This is Revolut Business API documentation. Here you will find what functionality is available to you to integrate your Revolut business account into your current and future business processes and to automate your business transactions. There are APIs for most of what you can do via your Revolut Business web portal.

    To help you navigate, this page is split into 3 vertical sections:

    • the navigation menu and search on the left,
    • the content in the middle,
    • and usage examples on the right.

    Getting started

    Pass your API-key in all API requests as a header

    GET /api/1.0/accounts HTTP/1.1
Authorization: Bearer <your api-key>

    Once you sign up for your Revolut Business account on business.revolut.com, you will be issued with two API-keys, one for sandbox and one for production environment. The sandbox environment is useful for development and allows to play with test data. Some of the data will be pre-populated so you can start quickly.

    Please DO NOT use production environment until you have tested your integration sufficiently and are happy that it works as expected.

    Payments

    All incoming and outgoing payments are represented as transactions and are processed in two stages from the user's perspective:

    1. a new transaction is created,
    2. the created transaction is processed, i.e.
      • credit/debit on both sides of transaction if Revolut-to-Revolut,
      • posted to the external payment network (Faster Paymetns, SEPA, SWIFT etc.).

    A new transaction has pending state, and processed transaction's state can be one of completed, failed, reverted or declined.

    Transfer between own accounts

    curl -X POST https://sandbox-b2b.revolut.com/api/1.0/transfer \
  -H "Authorization: Bearer <your api-key>" \
  --data @- << EOF

    {
  "request_id": "e0cbf84637264ee082a848b",
  "source_account_id": "bdab1c20-8d8c-430d-b967-87ac01af060c",
  "target_account_id": "5138z40d1-05bb-49c0-b130-75e8cf2f7693",
  "amount": 123.11,
  "currency": "EUR",
  "description": "Expenses funding"
}

    EOF

    Response

    {
  "id": "62b61a4f-fb09-4e87-b0ab-b66c85f5485c",
  "state": "completed",
  "created_at": "2017-06-21T11:22:11.0Z",
  "completed_at": "2017-06-21T11:22:11.0Z"
}

    This endpoint processes transfers between your accounts. Such transfers are normally processed instantly, however, it is possible to provide an optional webhook in case the transfer processing is delayed. The webhook should be able to process a POST request. If the provided endpoint if not available or fails (responds with 50x HTTP error) for any reason, the call will be retried 3 times with an increasing interval between attempts.

    The resulting transaction has type "transfer".

    Request

    POST https://sandbox-b2b.revolut.com/api/1.0/transfer

    Request fields

    Field Description Format
    request_id a unique value used to handle duplicates submitted as a result of lost connection or another client error (40 characters max) Text
    source_account_id the ID of a source account UUID
    target_account_id the ID of a target account UUID
    amount a amount to transfer Decimal
    currency a currency to transfer, both source and target accounts should be in this currency 3-letter ISO currency code
    description a textual reference shown on the transaction Text
    webhook an optional webhook that will be called on transaction completion URL

    Response

    Field Description Format
    id the ID of the created transaction UUID
    state the transction state: pending, completed, declined or failed Text
    created_at the instant when the transaction was created ISO date/time

  "created_at": "2017-10-01T10:10:10.0Z"

    This endpoint creates a new payment. If the payment is for another Revolut account, business or personal, the transaction may be processed synchronously.

    To avoid duplicate payment submission because on an error in your code, we strongly recommend to provide a unique request_id for each submitted payment. The request_id must be previously persisted on your side.

    Request

    POST https://sandbox-b2b.revolut.com/api/1.0/pay

    Request fields

    Field Description Format
    request_id the client provided ID of the transaction (40 characters max) Text
    account_id the ID of the account to pay from UUID
    receiver.counterparty_id the ID of the receiving counterparty UUID
    receiver.account_id an optional ID of the receiving counterparty's account, can be own account (only for internal counterparties) UUID
    amount the transaction amount Decimal
    currency the transaction currency Decimal
    description the transaction description Text
    webhook an optional webhook that will be called when transaction is processed URL

    Response

    Field Description Format
    id the ID of the created transaction UUID
    state the transction state: pending, completed, declined or failed Text
    reason an optional reason code for declined or failed transaction state Text
    created_at the instant when the transaction was created ISO date/time

    Schedule Payment

    curl -X POST https://sandbox-b2b.revolut.com/api/1.0/pay \
  -H "Authorization: Bearer <your api-key>" \
  --data @- << EOF

    {
  "request_id": "e0cbf84637264ee082a848b",
  "account_id": "bdab1c20-8d8c-430d-b967-87ac01af060c",
  "receiver": {
    "counterparty_id": "5138z40d1-05bb-49c0-b130-75e8cf2f7693"
  },
  "amount": 123.11,
  "currency": "EUR",
  "description": "Invoice payment #123",
  "webhook": "http://mega.company.com/transaction/e0cbf84637264ee082a848b",
  "schedule_for": "2017-10-10"
}

    EOF

    It is possible to schedule an internal payments for up to 30 days ahead. Scheduling external payments is not supported at the moment. Scheduled payment must be in currency of the account to pay from. To schedule a future payment, as opposed to immediate execution, just pass an additional field in the POST /pay request:

    Field Description Format
    schedule_for a future date/time ISO date/time
    Payments scheduled for the same time are not guaranteed to be executed all at the same time. As scheduled processor runs every several hours, the payments will be processed in batches, and if you schedule too many payments it will require more time to process all of them.

    This endpoint retrieves transaction details by transaction ID or by request ID.

    Request

    To retrieve a transaction by ID:

    GET https://sandbox-b2b.revolut.com/api/1.0/transaction/<id>

    To retrieve a transaction by request ID:

    GET https://sandbox-b2b.revolut.com/api/1.0/transaction/<request_id>?id_type=request_id

    Response

    Field Description Format
    id the ID of transaction UUID
    type the type of transaction Text
    request_id the client provided request ID Text
    state the transction state: pending, completed, declined or failed Text
    reason an optional reason code for declined or failed transaction state Text
    created_at the instant when the transaction was created ISO date/time
    updated_at the instant when the transaction was last updated ISO date/time
    completed_at the instant when the transaction was completed, mandatory for completed state only ISO date/time
    scheduled_for an optional date when the transaction was scheduled for ISO date
    webhook the webhook provided at transaction creation URL
    legs the legs of transaction, there'll be 2 legs between your Revolut accounts and 1 leg in other cases Array
    legs[n].leg_id the ID of the leg UUID
    legs[n].amount the transaction amount Decimal
    legs[n].currency the transaction currency Decimal
    legs[n].account_id the ID of the account the transaction is associated with Decimal
    legs[n].counterparty.id the counterparty ID Text
    legs[n].counterparty.account_id the counterparty account ID Text
    legs[n].counterparty.type the type of counterparty: self, revolut, external Text
    legs[n].description the transaction description, sometimes referred to as "payment reference" Decimal

    URL Parameters

    counterparty an optional counterparty id UUID
    count an optional number of records to return, default is 100 Number
    type an optional transaction type Text

    Errors

    This error section is stored in a separate file in `includes/_errors.md`. Slate allows you to optionally separate out your docs into many files...just save them to the `includes` folder and add them to the top of your `index.md`'s frontmatter. Files are included in the order listed.

    The Kittn API uses the following error codes:

    Error Code Meaning
    400 Bad Request -- Your request sucks.
    401 Unauthorized -- Your API key is wrong.
    403 Forbidden -- The kitten requested is hidden for administrators only.
    404 Not Found -- The specified kitten could not be found.
    405 Method Not Allowed -- You tried to access a kitten with an invalid method.
    406 Not Acceptable -- You requested a format that isn't json.
    410 Gone -- The kitten requested has been removed from our servers.
    418 I'm a teapot.
    429 Too Many Requests -- You're requesting too many kittens! Slow down!
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance. Please try again later.
